# Workflow Password Node Plan

This document defines the planned password hashing and password verification nodes for Workflow APIs. The goal is to make register/login workflow templates safe enough to use without storing plain passwords.

Ye plan auth workflows ke liye hai. Plain password database, workflow JSON, logs, history, webhooks, realtime events, ya docs examples me store nahi hona chahiye.

## Core Rule

Never store or return a plain password.

Allowed:

```json
{
  "passwordHash": "{{steps.hash-password.hash}}"
}
```

Not allowed:

```json
{
  "password": "{{body.password}}"
}
```

## Planned Nodes

Two nodes should be added.

### hashPassword

Purpose: accept a plain password from request body or another trusted source and produce a password hash.

Planned config:

```json
{
  "id": "hash-password",
  "type": "hashPassword",
  "config": {
    "password": "{{body.password}}",
    "outputKey": "passwordHash"
  }
}
```

Planned output:

```json
{
  "algorithm": "argon2id",
  "hash": "$argon2id$v=19$m=65536,t=3,p=1$...",
  "needsRehash": false
}
```

Only `hash`, `algorithm`, and `needsRehash` are safe to store. The plain input must be redacted from test output and run history.

### verifyPassword

Purpose: compare a plain password with a stored password hash during login.

Planned config:

```json
{
  "id": "verify-password",
  "type": "verifyPassword",
  "config": {
    "password": "{{body.password}}",
    "hash": "{{steps.find-user.entries.0.data.passwordHash}}",
    "outputKey": "passwordCheck"
  }
}
```

Planned output:

```json
{
  "matched": true,
  "needsRehash": false
}
```

The output must not include the plain password or the full stored hash unless a later implementation proves the hash is already available elsewhere in the workflow history. Safer default: output only `matched` and `needsRehash`.

## Algorithm

Preferred algorithm:

```txt
argon2id
```

Reason:

- It is designed for password hashing.
- It supports memory hardness.
- It has a strong security track record when parameters are configured correctly.

Fallback if native dependency constraints block Argon2:

```txt
scrypt
```

Fallback rules:

- Use Node `crypto.scrypt` or a well-maintained audited package.
- Store algorithm marker with the hash.
- Keep future migration path to Argon2id.

Do not use:

- Unsalted SHA-256.
- MD5.
- Reversible encryption for passwords.
- Static shared salt.
- Plain text.

## Parameters

Target Argon2id starting parameters:

```txt
memoryCost: 65536 KiB
timeCost: 3
parallelism: 1
hashLength: 32
```

These values must be benchmarked on low-resource deploys before final release. The implementation should expose conservative env overrides only if needed:

```txt
APIAGEX_PASSWORD_ARGON2_MEMORY_KIB
APIAGEX_PASSWORD_ARGON2_TIME_COST
APIAGEX_PASSWORD_ARGON2_PARALLELISM
```

MVP should reject unsafe low settings.

## Salt

Rules:

- Generate a unique random salt per password.
- Salt must be generated by the password hashing library or `crypto.randomBytes`.
- Salt is stored inside the encoded hash string when the algorithm supports it.
- Never reuse one global salt for all users.
- Never accept salt from request body.

Hinglish: har user ke password ke liye alag random salt hoga. User se salt nahi lena hai.

## Register Workflow Shape

Safe register flow:

```txt
routeTrigger
validateBody(email,password)
queryEntries(users by email)
branch user exists
hashPassword(body.password)
createEntry(users: email, passwordHash, status)
returnResponse(201)
```

The create entry mapping should store:

```json
{
  "email": "{{body.email}}",
  "passwordHash": "{{steps.hash-password.hash}}",
  "status": "active"
}
```

The create entry mapping must not store:

```json
{
  "password": "{{body.password}}"
}
```

## Login Workflow Shape

Safe login flow:

```txt
routeTrigger
validateBody(email,password)
queryEntries(users by email)
branch user exists
verifyPassword(body.password, stored passwordHash)
branch password matched
issueToken
returnResponse(200)
```

Failed login should return a generic error:

```json
{
  "ok": false,
  "error": "INVALID_CREDENTIALS"
}
```

Do not reveal whether the email exists or password was wrong.

## Timing And Comparison Safety

Password verification must use the password hashing library's verify function. It must not compare raw strings with `===`.

If fallback comparison is ever needed for fixed-length derived values, use timing-safe comparison from Node crypto. For password hashes, rely on the hashing library verify API.

Failure paths should be shaped so obvious user enumeration is avoided:

- Same response body for missing user and wrong password.
- Similar status code.
- Optional fake verify operation for missing user if timing difference is material.

## Output And Redaction

Sensitive inputs:

```txt
password
newPassword
confirmPassword
currentPassword
```

Redact from:

- Workflow test output.
- Workflow run history.
- Node error details.
- Server logs.
- Webhook payload previews.
- Realtime history.
- Admin UI debug panels.

Safe output examples:

```json
{
  "hash": "$argon2id$...",
  "algorithm": "argon2id",
  "needsRehash": false
}
```

```json
{
  "matched": false,
  "needsRehash": false
}
```

## Validation Rules

`hashPassword` validation:

- `password` is required.
- `password` must resolve to a string.
- Minimum password length should default to 8 or higher before hashing.
- `outputKey` is optional but must be a safe variable name if present.

`verifyPassword` validation:

- `password` is required.
- `hash` is required.
- Both must resolve to strings.
- Unsupported hash algorithm returns a controlled failure.

Planned error codes:

```txt
PASSWORD_INPUT_MISSING
PASSWORD_INPUT_TOO_SHORT
PASSWORD_HASH_FAILED
PASSWORD_HASH_UNSUPPORTED
PASSWORD_VERIFY_FAILED
PASSWORD_VERIFY_INVALID_HASH
```

## Migration Risk

Existing projects may already have user schemas with insecure password fields from early templates.

Migration guidance:

- Keep old placeholder docs visible and mark them unsafe for production.
- Add docs telling users to rename/use `passwordHash`, not `password`.
- Do not auto-migrate plain password fields.
- If a project already stored plain passwords, admins must reset user passwords.
- `needsRehash` lets future login workflows upgrade old hashes after successful verify.

Hash upgrade flow:

```txt
verifyPassword -> if matched and needsRehash -> hashPassword -> updateEntry(passwordHash)
```

## Admin UI Requirements

Workflow builder should expose:

- `Hash password` node.
- `Verify password` node.
- Password input picker with warning that the plain value is never stored.
- Output mapping helper for `passwordHash`.
- Redacted workflow test result.
- Template warning when a createEntry step maps `{{body.password}}` directly.

## Tests Required For Implementation

Minimum tests:

- Hash node returns a valid encoded hash and never returns the plain password.
- Same password hashes differently because salts differ.
- Verify node succeeds for matching password.
- Verify node fails for wrong password.
- Verify node handles malformed hash without throwing raw errors.
- Minimum password length is enforced.
- Test run output redacts password input.
- Register workflow template stores `passwordHash`, not `password`.
- Login workflow with missing user and wrong password returns the same public error.
- `needsRehash` is true when parameters are weaker than current policy.

## Implementation Order

1. Choose and add password hashing dependency after package review.
2. Add password hashing config defaults.
3. Add `hashPassword` node type and validator.
4. Add `verifyPassword` node type and validator.
5. Add redaction coverage for password fields.
6. Update register template to use `hashPassword`.
7. Add login template after issue-token plan is complete.
8. Add Admin UI node fields.
9. Add security-focused tests.

Do not ship user registration/login templates as production-ready until password hashing and token issuing are both implemented.
